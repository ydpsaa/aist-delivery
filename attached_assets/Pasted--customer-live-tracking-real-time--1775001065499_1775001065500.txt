Отлично. Теперь переходим к следующей подзадаче: реализовать customer live tracking и real-time обновления статуса заказа для клиента внутри мобильного приложения.

Что нужно сделать сейчас:
1. Добавить real-time канал для customer app.
2. Клиент должен получать live updates по своему заказу без ручного refresh.
3. Использовать тот же общий подход, который уже применён для courier mode:
   - WebSocket
4. Не ломать текущий courier flow.
5. Не убирать push notifications.
6. Сделать customer live tracking как отдельный, чисто изолированный слой.

Что должно работать после этого этапа:
1. Когда клиент создаёт заказ, он может открыть tracking screen и видеть live status.
2. Когда заказ получает курьера, статус обновляется instantly.
3. Когда курьер:
   - принимает заказ
   - приезжает
   - забирает заказ
   - доставляет заказ
   клиент видит это мгновенно на tracking screen.
4. Если заказ отменён, клиент тоже instantly видит обновление.
5. Если приложение активно и открыт tracking screen, live updates должны приходить без polling.
6. Если приложение в фоне или закрыто — push notifications остаются fallback-механизмом.

Backend requirements:
1. Добавить отдельный customer WebSocket flow.
2. Реализовать customer event channel:
   - emitToCustomer(customerId, event)
3. Добавить customer WebSocket endpoint:
   - /api/customer/ws
4. Проверять JWT и роль customer на этом канале.
5. Backend должен отправлять customer events на каждом ключевом этапе order lifecycle:
   - order_created
   - courier_assigned
   - courier_arrived
   - picked_up
   - delivered
   - cancelled
6. Все events должны выходить из уже существующей order lifecycle логики, а не быть разбросаны хаотично.
7. Не ломать текущий courier event bus — расширить архитектуру аккуратно.

App requirements:

4. При получении events обновлять tracking UI instantly.
5. Добавить нормальные loading / connecting / reconnect states.
6. Не перегружать UI.
7. Не менять без причины уже готовые customer screens, кроме tracking integration.
